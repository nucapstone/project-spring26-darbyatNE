## Project Evolution: Electrcity Price Transparency Dashboard

The core objective of this project has shifted from expanding predictive features to expanding geographical and architectural scale. The client prioritized a system capable of handling additional service territories over a localized forecasting model. 

Here is a high-level comparison of how the project scope has evolved to meet this new mandate:

| **Project Phase** | **Primary Focus** | **Details** |
|-------------------|-------------------|---------------------------------|
| **Initial Plan** | Predictive Modeling | Forecast retail standard offer rates using aggregated wholesale LMPs, natural gas forward prices, capacity prices, and RECs. |
| **The Pivot** | Architectural Scale | Expand the system's capacity from 46 initial companies to potentially 3,000+ electric service companies across different ISO/RTOs. |
| **Revised Plan** | Spatial Data Engineering | Apply GeoJSON parsing, RESTful APIs, pandas, and Postgres spatial joins to bridge the gap between CEEPR economic retail data and raw EIA/PJM wholesale grid rates. |

## Challenges Encountered & Engineered Solutions

Building a scalable system from scratch revealed several unexpected roadblocks, particularly regarding data access and the lack of existing "ground truth" mapping. Here is how those challenges were resolved:

### 1. Aligning with Stakeholder Priorities
- **The Problem:** The initial focus on classifying and predicting future electric rates was intellectually interesting, but it did not align with the stakeholder's immediate need for a scalable infrastructure.
- **The Solution:** Pivoted the project focus entirely to data engineering and architecture, demonstrating the flexibility required to deliver actual stakeholder value.

### 2. Overcoming Restricted Infrastructure Data
- **The Problem:** High-quality electrical infrastructure data that was freely available a few years ago is now highly restricted. The Department of Homeland Security currently limits access to this data exclusively to US government employees.
- **The Solution:** Sourced a complete, open-access file system via University of Michigan online resources that is only two years old, bypassing the federal restriction while maintaining data integrity.

### 3. Bridging the "Ground Truth" Gap
- **The Problem:** Retail Service Territory source prices are essentially a black box. There is no existing ground truth linking specific wholesale pricing points (LMPs) to specific retail service territories.
- **The Solution:** Engineered a custom mapping solution by combining three distinct data sources:
  1. **PJM Dataminer 2:** Extracted node names, pnode IDs, and hourly prices.
  2. **Overpass API:** Queried OpenStreetMap (OSM) layers for electrical substations, yielding names and exact X/Y coordinates in a GeoJSON format.
  3. **Spatial Joining:** Compared these X/Y coordinates against distinct areas of CEEPR interest using the UMich National Service Territory Boundary Map to successfully group LMPs to their respective retail territories.

### 4. Navigating PJM Price Data Restrictions
- **The Problem:** PJM enforces strict API limits: maximum 6 queries per minute, 50,000 rows per query, and data older than two years is archived (meaning it cannot be queried with standard filters).
- **The Solution:** Built a smart, rate-limiting Python script. 
  - The script dynamically pauses if the time since the last query is under 10 seconds.
  - It includes a data checker that splits the query method depending on whether the target data is archived or active.
  - Reduced the total number of hourly LMPs queried from 13,500 down to 705. This optimization maintains spatial variety without overwhelming the system with unnecessary volume.

## What Lies Ahead

With the data capture methodology successfully established for the PJM area service territories of interest to CEEPR, the project now moves into the final synthesis and visualization phase. 

The three primary tasks remaining are:

1. **Data Aggregation:** Transform and aggregate the raw wholesale data into a structured format that is perfectly compatible with the existing retail dataset.
2. **UI Development:** Complete the user interface (featuring interactive maps and line charts) to allow users to select specific time periods, visually evaluate the data, and discover pricing trends.
3. **Scale Documentation:** Write comprehensive documentation detailing this exact methodology so the client can easily expand the number of service territories being tracked for CEEPR in the future.
